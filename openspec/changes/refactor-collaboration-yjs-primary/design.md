## Context

当前编辑器前端实际运行的是旧 WebSocket 协同 store，Yjs store 仍停留在原型状态；后端实际接线的也是旧协同服务，而不是 `services/y-collab/collab-basic-service.js`。与此同时，编辑页的评论、历史、画布和导出能力都直接或间接依赖旧协同 store，所以这次切换不是简单替换 transport，而是一次跨前后端、跨 UI 入口的主线收口。

用户已经明确第一轮目标：

- 用 Yjs 成为唯一协同主线；
- 只先打通块配置、页面配置和在线状态；
- 默认进入个人持久房间；
- 支持手动输入房间名进入共享持久房间；
- 共享房间不能直接发布；
- 评论、历史、画布协同和依赖协同状态的导出能力第一轮先隐藏或禁用。

## Goals / Non-Goals

**Goals:**

- 在不改变编辑页核心使用方式的前提下，把现有协同入口统一切到 Yjs。
- 为每个登录用户提供稳定的个人持久房间，支持跨设备恢复到同一份页面工作副本。
- 支持基于页面和房间名的共享持久房间，且首次创建时使用创建者当前个人房间内容作为种子。
- 让发布流程只在个人房间中可用，并把当前 Yjs 文档状态写回页面数据库。
- 移除第一轮不支持能力的 UI 暴露，避免用户进入不完整流程。

**Non-Goals:**

- 第一轮不迁移评论、历史、画布协同到 Yjs。
- 第一轮不支持共享房间直接发布、分享链接、邀请码或精细权限控制。
- 第一轮不做匿名协同，协同房间统一要求登录身份。
- 第一轮不实现共享房间导出或将共享房间状态直接作为正式页面内容。

## Decisions

### 1. 保留现有前端 store 引用入口，但把实现切到 Yjs

前端大量组件和 store 都已经直接依赖 `src/stores/collaborationStore.ts`。第一轮不改业务方 import 路径，而是把该入口收口成 Yjs 主实现，避免大范围改动组件依赖树。

备选方案：

- 直接把所有引用改到 `src/stores/CRDT/collaborationStore.ts`：更“纯”，但需要同时重写大量调用点，第一轮噪音太大。

### 2. 使用两类工作区键：个人房间和共享房间

房间键统一建立在 `pageId` 上：

- 个人房间：`personal:<pageId>:<username>`
- 共享房间：`shared:<pageId>:<roomName>`

这样个人开发与共享协作的边界清楚，且能满足“自己长期开发，也能手动加入别人的房间协同”。

备选方案：

- `pageId` 直接唯一对应一个房间：无法同时满足个人开发与共享协同。
- 绑定本地设备 ID：无法跨设备进入同一份个人房间。

### 3. 新房间的种子数据按房间类型分别确定

- 个人房间首次不存在时，用页面数据库当前保存内容初始化。
- 共享房间首次创建时，用创建者当前个人房间内容初始化。

这样个人房间可以代表用户自己的长期工作副本，共享房间则代表一次从个人工作副本派生出来的协作分支。

备选方案：

- 新房间统一空白初始化：用户体验差，且容易丢失上下文。
- 共享房间统一从页面数据库初始化：与“基于当前个人版本发起协同”的诉求不符。

### 4. 发布只允许发生在个人房间

个人房间代表用户的正式工作副本；共享房间代表协作分支。第一轮中，只有个人房间允许发布，发布时读取当前 Yjs 文档状态并回写页面数据库；共享房间的发布入口直接禁用。

备选方案：

- 允许共享房间直接发布：责任边界不清，且会让共享协作副本直接覆盖正式页面内容。

### 5. 第一轮先隐藏或禁用非核心协同能力

评论、历史、画布协同以及依赖协同服务状态的导出能力，当前都深度绑定旧 store。如果第一轮强行保留，会把一次主线切换扩大成全面迁移。第一轮应直接隐藏或禁用这些入口，让页面暴露出的能力与实际实现保持一致。

备选方案：

- 保留 UI 但只给提示：仍会制造大量半可用状态。
- 一起迁移所有能力：范围明显超出第一轮。

### 6. 服务端持久化使用 Yjs 文档快照作为单一真相

服务端以 Yjs 文档状态作为个人房间和共享房间的持久化真相，连接时加载，文档更新时增量保存，用户全部离开后再做最终保存与内存清理。页面数据库仅在首次个人房间初始化和个人房间发布时参与同步。

备选方案：

- 同时维护页面数据库和旧协同内存结构作为运行真相：会继续保留双轨数据源。

## Risks / Trade-offs

- [风险] 现有 Yjs 服务端原型代码不稳定，存在无法直接接入的问题 → 缓解：第一步先修正 Yjs 服务端和持久化接口，再切换 server 入口。
- [风险] 现有大量组件调用旧 store 的评论/历史/画布 API → 缓解：第一轮统一隐藏或禁用相关入口，而不是保留空实现。
- [风险] 个人房间绑定用户名，若用户名可变会影响个人房间定位 → 缓解：第一轮明确用户名作为稳定键使用；若未来支持用户名修改，再补迁移策略。
- [风险] 共享房间禁发版可能让部分协作流程多一步切回个人房间 → 缓解：退出共享房间后自动回个人房间，降低切换成本。

## Migration Plan

1. 修正并增强 Yjs 服务端，使其能稳定处理房间连接、加载、持久化和快照种子。
2. 将服务端协同入口从旧 WebSocket 服务切换到 Yjs 服务。
3. 将前端 `collaborationStore` 入口替换为 Yjs 实现，并保留现有业务方 import 路径。
4. 重写房间模型，使编辑页自动进入个人房间，并支持手动切换到共享房间。
5. 下线第一轮不支持的协同 UI 和交互。
6. 将发布流程改为从当前个人房间的 Yjs 状态回写页面数据库。

回退策略：

- 若切换后出现阻断性问题，可回退到旧协同服务接线和旧 `collaborationStore` 实现；
- 因为第一轮不删除旧实现文件，回退成本主要是恢复入口和 UI 开关。

## Open Questions

- 第一轮不处理共享房间内容回流到个人房间的显式“同步回我”能力，后续是否需要补一个从共享房间复制回个人房间的动作，待第二轮再决定。
