## 目标范围

本计划实现“Yjs 成为唯一协同主线”的第一轮落地，严格限制范围为：

- 核心协同：块配置、页面配置、在线状态
- 工作区模型：个人持久房间、共享持久房间
- 发布约束：仅个人房间可发布
- UI 收口：隐藏或禁用未迁移的评论、历史、画布协同和依赖协同数据的导出入口

不在本轮实现：

- 评论、历史、画布协同迁移
- 共享房间直接发布
- 邀请链接、邀请码、细粒度权限
- 匿名协同

## 模块职责映射

- `y-collab/collab-basic-service.js`
  负责 Yjs 房间创建、持久化加载、共享房间种子初始化、在线状态和房间切换相关服务端逻辑。
- `collaborationStore.ts`
  作为前端唯一协同入口，负责个人/共享房间连接、块配置与页面配置同步、当前工作区元信息和在线人数。
- `roomStore.ts`
  负责房间切换语义，不承载 Yjs 协议细节。
- `edit.ts`
  负责把本地编辑状态与协同层对接，并防止远程同步导致本地 watch 循环。
- `edit-header.vue` / `collabModel.vue`
  负责房间入口、共享房间创建/加入、发布禁用状态和当前房间提示。
- 评论/历史/画布相关组件
  第一轮只做 UI 收口，避免继续调用旧协同 API。

## 任务顺序

### 任务 1：修正并接通 Yjs 服务端主线

目标：

- 让服务端能够稳定创建和恢复个人/共享房间
- 按房间键持久化 Yjs 文档状态

要改的文件：

- `packages/moten-server/services/y-collab/collab-basic-service.js`
- `packages/moten-server/services/collab/collab-storage.js`
- `packages/moten-server/index.js`

验证：

- 服务端能启动
- 个人房间首次进入能从页面数据库内容初始化
- 断开后重新连接能恢复到同一房间状态

### 任务 2：用 Yjs 重写前端统一协同入口

目标：

- 保持 `useCollaborationStore()` 调用方式不变
- 内部改成 Yjs 实现

要改的文件：

- `packages/moten-editor/src/stores/collaborationStore.ts`
- `packages/moten-editor/src/stores/roomStore.ts`
- `packages/moten-editor/src/stores/edit.ts`
- `packages/moten-editor/src/types/collab.ts`

验证：

- 打开编辑页自动进入个人房间
- 块配置和页面配置在两个客户端间同步
- 在线人数状态正常更新

### 任务 3：实现个人/共享房间交互

目标：

- 默认个人房间
- 手动输入房间名创建/加入共享房间

要改的文件：

- 进入已有共享房间时恢复共享状态
- 退出共享房间时自动回个人房间

- 只有个人房间允许发布

- `packages/moten-editor/src/components/edit/edit-header.vue`

验证：

- 个人房间可以正常发布

- 发布后重新打开个人房间，内容与发布结果一致

## 风险点

- 现有 Yjs 服务端原型代码明显不稳定，可能需要先修语法/接口问题才能继续。
- 旧 store API 面较大，若直接删除会连带大量组件报错；第一轮应优先保持入口兼容。
- 页面数据库内容与 Yjs 文档之间的首轮初始化、发布回写是双向边界，最容易出现状态不一致。
- 评论/历史/画布相关组件分散，漏掉入口就会留下坏链路。

```powershell

```

## 回退思路

- 保留旧 `services/collab/*` 和旧协同协议文件，第一轮只移除业务入口，不立即物理删除。
- 若 Yjs 主线切换失败，可先恢复 `packages/moten-server/index.js` 对旧协同服务的接线。
- 前端若出现大面积回归，可先恢复 `src/stores/collaborationStore.ts` 到旧实现，再分步引入 Yjs 能力。
